# Critic MEMORY

## このプロジェクトで見落としやすいリスク

- **廃止フロー残存**: 設計変更時に旧モジュール（decomposer 等）が `#[allow(dead_code)]` で残りがち。
  呼び出し元がないだけで API エンドポイント等から間接参照されているケースを見落としやすい。
  → mod.rs だけでなく server/ 等のユーザー側コードも必ず確認する。

- **session_id の二重判定**: `has_session` フラグと `session_id.is_some()` を別々の場所で算出するパターンは
  将来の乖離リスクになる。1箇所で判定する設計かどうか確認する。

## 計画品質の判定基準

- フロー定義が DB クエリ（get_XXX_task）に対応しているか 1:1 で確認する
- `--resume` の繋ぎ目（session_id 保存 → 受け渡し → フラグ渡し）の3点を個別に追う
